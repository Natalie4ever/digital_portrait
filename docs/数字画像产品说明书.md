# 员工数字画像系统 — 产品说明书

> **版本**：v1.0（基于 2026-03-23 代码库）
> **形态**：内网单机部署的 PC 端 Web 系统（前后端分离）
> **技术栈**：FastAPI + SQLAlchemy(async) + SQLite / React 18 + Vite + Ant Design
> **目标用户**：商业银行内部的人力资源 / 党务 / 员工关怀等岗位
> **项目代号**：`digital_portrait`（员工数字画像）

---

## 一、产品定位与解决的核心问题

### 1.1 产品定位

员工数字画像系统是面向"集约运营中心（广东）"等内网机构员工的**全维度电子档案与家访管理平台**。它把过去散落在 Excel、纸质表、邮件里的员工信息，集中到一套角色化、可检索、可导出、可审计的 Web 系统里，让管理层、组长、员工本人能在统一入口完成"看档案—改档案—做家访—查日志"全流程。

### 1.2 解决的核心问题（业务痛点）

| # | 痛点 | 系统如何解决 |
|---|------|------------|
| 1 | **员工档案散落、版本不一致**（基础信息、政治面貌、学历、家庭、简历、奖惩、资格、成果、语言、通讯、技能标签等多类信息分布在多个 Excel） | 统一建模为 `profiles` 主表 + 8 个子表，所有信息一条记录一次写入、永不丢失 |
| 2 | **EHR 号格式不统一**（有的带前导零、有的写错位数），导致跨系统对不上 | 全局校验：EHR 号必须**恰好 7 位数字**，登录、查询、批量导入处统一拦截 |
| 3 | **手工维护家庭/学历/简历多版本记录效率低** | 子表全部支持**多条、按时间排序**（学历按 `enrollment_date`，简历按 `start_time`，奖惩/资格/成果按 `obtain_time`），并提供行内新增/修改/删除 |
| 4 | **家访记录以纸质表为主，查找、统计、归档困难** | 完整电子化家访：组长在线填写"家访明细+家访记录表"两段信息，支持**预览打印、导出 PDF、批量导出 Excel** |
| 5 | **家访对象有组别边界，不能跨组家访，但纸质流程无法校验** | 组长账号绑定"有效组别"（`leader_effective_group`），后端硬性校验 `visited_user.group_name == viewer.group_name`，越权直接 403 |
| 6 | **角色权限混乱**：普通员工能看别人档案、组长能改他人档案 | 三级角色 + 资源级 ACL：本人/本组/全员，按接口分别判定 `_can_access` / `_can_edit` / `_can_delete` |
| 7 | **敏感操作无可追溯**（谁改了谁、谁导入了哪些用户、谁删了家访） | 全量 `operation_logs` 表，关键操作（登录、改密、用户增删改、批量导入、家访 CRUD）100% 落库，管理员页可查 |
| 8 | **技能标签无人管理**，员工自填五花八门 | 双层结构：管理员维护**预定义模板** + 员工在档案里**选择模板或自定义**，并防重复 |
| 9 | **身份证/手机号/座机/日期数据脏**，后期做合规、报表困难 | 后端 `validators.py` 统一强校验：18 位身份证 + 校验码、1[3-9]xxxxxxxxx、座机 7-20 位、日期多格式解析 |
| 10 | **1 小时无操作后敏感页面长期挂着**（内网共享电脑场景） | 后端 JWT 60 分钟过期 + 前端 60 分钟 idle 检测 + 401 自动跳登录页 |

## 二、用户与角色模型

### 2.1 角色矩阵

| 角色 | 简称 | 主要场景 | 关键边界 |
|------|------|----------|----------|
| **普通员工** | `user` | 维护自己档案、查看自己家访 | 看不到任何他人数据；不能进家访新建/编辑 |
| **组长** | `leader` | 查看本组成员档案、新建/编辑本组成员家访 | 跨组 0 权限；未配置组别时连列表都进不去 |
| **管理员** | `admin` | 全员档案、用户管理、操作日志、技能标签模板、家访**只读** | 不能编辑家访（业务规则：仅组长可编辑） |

### 2.2 权限规则（核心判定）

| 资源 | 普通员工 | 组长 | 管理员 |
|------|----------|------|--------|
| 我的档案 | 查看+编辑 | 查看+编辑 | 查看+编辑 |
| 他人档案 | ❌ 403 | 同组：查看 / 异组：❌ | ✅ 查看+编辑基础信息 |
| 档案管理列表 | ❌ | 仅本组 | 全员 |
| 用户管理 | ❌ | ❌ | ✅ |
| 操作日志 | ❌ | ❌ | ✅ |
| 技能标签模板（读） | ✅ | ✅ | ✅ |
| 技能标签模板（写） | ❌ | ❌ | ✅ |
| 家访列表（看） | 仅自己 | 仅本组 | 全员 |
| 家访详情（看） | 仅自己 | 同组 | 全员 |
| 家访新建 | ❌ | ✅ 仅本组 | ❌ |
| 家访编辑 | ❌ | ✅ 仅本组 | ❌（接口兜底 403） |
| 家访删除 | ❌ | ✅ 仅本组 | ✅ 任意 |
| 家访批量导出 Excel | ❌ | ✅ 可见范围内 | ✅ 全部 |

## 三、功能模块清单

> 数字编号与左侧导航顺序一致。

### M1. 登录与认证

**目的**：安全、便捷、可追溯地进入系统。

- **登录入口**：[Login.jsx]，单字段：EHR 号（7 位数字）+ 密码
- **EHR 预校验**：输入 7 位后**失焦/输入完成自动**回查 `/api/auth/check-ehr/{ehr_no}`，命中则提示员工姓名；密码框**始终可见**（不随预校验切换）
- **登录接口**：`POST /api/auth/login` → 颁发 JWT（HS256，60 分钟过期），登录行为写 `operation_logs`
- **失败提示**：账号/密码错 → "EHR号或密码错误"；账号禁用 → "账号已禁用"
- **修改密码**：`POST /api/auth/change-password`，需提供原密码；成功后新密码可立即登录
- **当前用户**：`GET /api/auth/me` → `{ehr_no, name, group_name, role}`，前端全局 `AuthContext` 持有
- **自动登出**：前端 `Layout` 监听 click/keydown，60 分钟无活动 → 清 token + 跳 `/login`；后端 60 分钟 token 过期 → 401 → 前端拦截跳登录

### M2. 我的档案（核心模块）

**目的**：员工自助维护一个"完整、可信、永远不丢"的电子档案。

页面路径：`/profile`，子表全部支持"行内新增 / 行内编辑 / 行内删除"，所有改动调用同一组后端 REST 接口，**保存无刷新**。

| 子模块 | 字段 | 后端路径 |
|--------|------|----------|
| **基础信息** | 性别、民族、出生日期、岗位、证件类型/号码、籍贯/出生地/户籍地（省市区级联）、参加工作日期、入职日期、婚姻状况 | `PUT /api/profile/me/base` |
| **政治面貌**（多条） | 政治面貌（群众/共青团员/中共党员/民主党派/其他）、加入日期、介绍人 | `/me/political` |
| **学历学位**（多条） | 教育类别（全日制/在职）、教育类型（国民/党校/军队）、学历（按毕/肄业细分 18 档）、学位（学士/硕士/博士/名誉博士）、毕业学校、专业、学制、入学/毕业日期、完成情况、授予国家 | `/me/education` |
| **家庭关系**（多条） | 姓名、性别、关系（父亲/母亲/配偶/子女等 8 档）、出生日期、工作单位及职位、政治面貌、人员状况（学前/上学/在业/无业/离退/已故/烈士/不详） | `/me/family` |
| **简历**（多条） | 起止时间、单位及职位 | `/me/resume` |
| **奖惩**（多条） | 奖惩类型（奖励/惩罚）、时间、名称、原因 | `/me/reward` |
| **外部资格**（多条） | 资格证书名称、获取时间 | `/me/qualification` |
| **专业成果**（多条） | 成果名称、获得时间 | `/me/achievement` |
| **语言能力**（多条） | 语种、熟练度（精通/熟练/良好/一般）、证书级别/分数 | `/me/language` |
| **通讯信息**（单条 upsert） | 移动电话、办公电话、家庭电话、家庭地址、邮箱、通勤时间（分钟） | `/me/contact` |
| **技能标签**（多条） | 标签名 + 可选 template_id 关联预定义模板 | `/me/skill-tags` |

**典型交互**：
- 子表使用 Ant Design `Table` + `Modal` 实现"行内编辑/删除"
- 表单校验全部走前端 AntD `rules` + 后端 Pydantic 双保险
- 保存后**无刷新**（局部 setState，不重拉整张表）

### M3. 档案管理（管理员/组长）

**目的**：跨用户/跨组检索档案，是 HR、党务、纪检的"找员工"工具。

- **入口**：`/admin/profiles`（仅 admin / leader）
- **筛选维度**：EHR 号、姓名、组别、角色、**技能标签**、**通勤时间阈值**、是否含已禁用
- **数据范围**：组长自动限定为 `group_name == leader.group_name`
- **列表展示**：EHR、姓名、组别、角色（彩色徽标）、技能标签（彩色 Chip）、通勤时长
- **详情查看**：点击行 → 跳转 `/admin/profile-view/:ehr_no`，复用 `Profile` 组件（但只读模式）

### M4. 用户管理（仅管理员）

**目的**：把控系统准入：谁能进、以什么身份进、什么时候不让进。

| 子功能 | 说明 |
|--------|------|
| **用户列表** | 列表 + 筛选（EHR、姓名、组别、角色、是否含已禁用）+ 分页；状态徽标：正常/已禁用 |
| **新增用户** | 弹窗表单：EHR、姓名、组别、角色、初始密码（不填则用默认 `1234567`） |
| **编辑用户** | 改姓名/组别/角色/启用禁用 |
| **删除用户** | 软删除（`deleted_at` 置时间），保留审计痕迹 |
| **重置密码** | `POST /api/admin/users/reset-password`，管理员可强改 |
| **批量导入** | 上传 `.xlsx`，列必须含**姓名、ehr号、组别**（可选角色/初始密码）；EHR 非法跳过、EHR 已存在 skip、合法创建；返回 `{created, skipped, errors}` |
| **Excel 模板** | 项目根目录 `批量导入用户测试.xlsx` 提供样例 |

### M5. 操作日志（仅管理员）

**目的**：审计与回溯。

- 路径：`/admin/logs`
- 来源：登录、改密、用户增删改、批量导入、家访 CRUD、子表 CRUD 等**所有写操作**自动落 `operation_logs`
- 字段：操作人（EHR + 姓名）、操作类型（action）、资源（resource）、详情（detail）、IP、时间
- 筛选：按用户 ID、按 action 模糊

### M6. 技能标签模板（仅管理员维护）

**目的**：让员工档案里的技能标签"既能选预定义又能自定义"。

- **读**：所有登录用户可在档案页 `Select` 下拉里看到全部模板
- **写**：仅管理员可在 `/admin/skill-tags` 增删模板（重名校验、不允许空名）
- **档案侧**：`profile_skill_tags` 表每行带 `template_id`（可选），删除模板不影响历史已用标签

### M7. 家访记录（业务特色模块）

**目的**：把"员工家访"从纸质表 + Excel 邮件的流程，搬到线上表单 + 一键导出。

#### 7.1 列表页 `/home-visits`

- **布局**：筛选条 + 表格 + 底部分页
- **筛选**：被家访人 EHR、家访年度、家访方式（线上/线下）、是否已家访
- **表格列**：被家访人（姓名 + EHR）、岗位、家访年度、家访时间、家访方式 Tag、是否已家访 Tag
- **操作列**（按角色动态显示）：
  - 查看 → `/home-visits/:id`（所有人）
  - 编辑 → `/home-visits/:id/edit`（**仅组长**）
  - 删除 → 二次确认（**组长+管理员**）
  - **批量导出明细表**：勾选/不勾选都支持，不勾选则导出当前页所有记录，输出 `家访明细表_YYYY-MM-DD_HHmm.xlsx`

#### 7.2 新建/编辑 `/home-visits/new` 与 `/home-visits/:id/edit`

- **Step 1（家访明细）**：
  - 被家访人下拉（**仅组长可访问**，且只能选本组成员）
  - 选择被家访人后**自动从档案回填**：岗位、移动电话、家属姓名/关系/工作单位
  - 家访年度（近 10 年）、家访时间、家访方式、家访地址、家访人员及岗位、是否已家访 Switch
- **Step 2（家访记录表）**：
  - 团队名称固定为"审核处理团队"（前端 disabled、后端校验）
  - 家访日期、地址、手机号、家庭电话
  - 家属 1 / 家属 2 各 4 字段（姓名/关系/联系方式/工作单位）
  - 员工家庭情况及家属反馈意见（TextArea，**预填"七点模板"**）：
    > （一）家庭主要成员情况：
    > （二）员工八小时以外动向：
    > （三）员工思想动态：
    > （四）家庭经济状况：
    > （五）家庭关系情况：
    > （六）员工家庭生活实际困难：
    > （七）总体评价及跟进意见：

#### 7.3 详情 `/home-visits/:id`

- 卡片式 `Descriptions` 展示"家访明细" + "家访记录表"两段
- 按钮组：**预览打印** / **导出 PDF**（仅组长可编辑）
- 预览打印：弹窗还原纸质表格，新窗口调用 `window.print()`
- 导出 PDF：`html2pdf.js` 把还原好的"集约运营中心（广东）员工家访记录表"渲染为 A4 PDF，文件名 `家访记录_姓名_日期.pdf`

#### 7.4 权限与边界

| 角色 | 列表 | 详情 | 新建 | 编辑 | 删除 | 导出 |
|------|------|------|------|------|------|------|
| 普通员工 | 仅自己 | 仅自己 | ❌ | ❌ | ❌ | 自己 |
| 组长 | 本组 | 本组 | 本组 | 本组 | 本组 | 可见 |
| 管理员 | 全部 | 全部 | ❌（接口兜底 403） | ❌ | 全部 | 全部 |

> 业务说明：管理员故意被设计为"只读"，以保证"谁做的家访，谁签字"——家访记录的责任主体是组长。

## 四、数据库概览

```
users                       用户主表（EHR 号、姓名、组别、角色、密码哈希、是否禁用、软删时间）
└── profile (1:1)           基础信息
    ├── political_info (1:N)  政治面貌
    ├── education_info (1:N)  学历学位
    ├── family_info (1:N)     家庭关系
    ├── resume_info (1:N)     简历
    ├── reward_info (1:N)     奖惩
    ├── qualification_info(1:N) 外部资格
    ├── achievement_info(1:N) 专业成果
    ├── language_info (1:N)   语言能力
    ├── contact_info (1:1)    通讯
    └── profile_skill_tags(N:M) → skill_tag_templates

home_visit_records           家访记录（被家访人/家访人/家访年度/家访时间/家访方式/家访地址/家访人员/是否已家访/团队/家访日期/岗位/联系电话/地址/手机号/家庭电话/家属1/家属2/反馈意见）

operation_logs               操作日志
```

字段类型全部使用 SQLAlchemy 通用类型（String/Date/DateTime/Boolean），**当前 SQLite，未来改 `DATABASE_URL` 即可平滑迁移到 SQL Server**。

## 五、技术架构

### 5.1 整体架构

```
浏览器 (React + Ant Design)
        │  /api/*   (Vite dev proxy → 8000)
        ▼
FastAPI (async)  ──  SQLAlchemy async  ──  SQLite (digital_portrait.db)
        │
        └─ JWT 鉴权 + 角色装饰器（get_current_user / get_current_admin）
        └─ 操作日志自动埋点（log_operation）
        └─ 业务校验（validators.py）
```

### 5.2 后端分层

| 层 | 文件 | 职责 |
|----|------|------|
| 入口 | `app/main.py` | FastAPI 实例、CORS、路由注册、启动时建表 |
| 配置 | `app/config.py` | 数据库 URL、JWT 密钥与过期、默认密码 |
| 认证 | `app/auth.py` | 密码哈希（PBKDF2-SHA256, 10 万次）、JWT 签发/解码、`get_current_user` 依赖、`leader_effective_group` 工具 |
| 数据库 | `app/database.py` | 异步引擎、Session 工厂、`Base` |
| 模型 | `app/models.py` | 12 张表的 SQLAlchemy ORM 定义 |
| 校验 | `app/validators.py` | EHR / 身份证（含校验码）/ 手机号 / 座机 / 日期统一工具 |
| Schema | `app/schemas.py` | Pydantic 请求/响应模型，含 `visited_ehr_no` 7 位校验 |
| 路由 | `app/routers/*.py` | 6 个领域路由：auth / users / profile / skill_tags / operation_logs / home_visits |
| 日志 | `app/operation_log.py` | 写操作埋点工具 |
| 迁移 | `backend/migrate_*.py` | 老库列扩展脚本（已存在的数据库手动跑一次） |

### 5.3 前端分层

| 层 | 文件 | 职责 |
|----|------|------|
| 入口 | `App.jsx` | 路由表 + 主题色（玫红 `#D05A6E`）+ 私有路由守卫 |
| 上下文 | `contexts/AuthContext.jsx` | 全局 `user`、`refreshUser`、`logout` |
| 布局 | `components/Layout.jsx` | 侧边栏菜单（按角色动态渲染）+ 顶栏 + 60 分钟 idle 登出 |
| API | `src/api.js` | `fetch` 封装、自动带 token、401 拦截、统一错误提示 |
| 常量 | `src/constants.js` | 民族/学历/学位/关系/奖惩/语言等所有下拉选项 |
| 页面 | `src/pages/*.jsx` | 13 个页面（Login/Home/Profile/ChangePassword/AdminUsers/AdminProfiles/AdminProfileView/AdminLogs/AdminSkillTags/HomeVisits/HomeVisitForm/HomeVisitDetail） |
| 导出组件 | `components/HomeVisitExport.jsx` | 纸质家访记录表的 HTML/CSS 还原，被详情页 + 预览打印 + PDF 导出三处复用 |

### 5.4 安全设计

| 维度 | 措施 |
|------|------|
| 密码存储 | PBKDF2-HMAC-SHA256 + 32 字节随机盐 + 10 万次迭代，存 `salt:hash` 形式 |
| Token | HS256 JWT，60 分钟过期；`Authorization: Bearer …` 头 |
| 密码字段 | 后端返回 `UserResponse` 不含 `password_hash`；批量导入不上传明文密码外的其它密码字段 |
| 软删除 | 用户 `deleted_at`；查询统一带 `deleted_at IS NULL` |
| 角色守护 | `get_current_admin` 装饰器、路由级 `current_user.role` 校验、资源级 `_can_access`/`_can_edit` |
| 跨组隔离 | 组长列表/查看/编辑/删除均以 `group_name` 强校验 |
| 数据校验 | EHR、身份证（含校验码）、手机号、座机、日期全部后端强校验 |
| 自动登出 | 前端 60 分钟 idle + 后端 60 分钟 token 过期双重保险 |
| 操作审计 | 所有写操作落 `operation_logs`，可按用户/动作检索 |

## 七、关键业务流程

### 7.1 首次使用流程

```
管理员跑 init_db.py
   → 创建默认管理员 EHR=0000001 / pwd=1234567
   → 登录
   → 批量导入用户（EHR 必填、姓名/组别必填）
   → 通知员工用 EHR + 默认密码首次登录
   → 员工到"我的档案"自行改密 + 补全档案
```

### 7.2 组长做家访流程

```
组长登录 → /home-visits → 新建家访记录
   → Step 1: 选本组组员（自动从档案带出岗位/电话/家属）
   → 填写家访年度/时间/方式/地址/家访人员/是否已家访
   → Step 2: 团队名称（自动 = 审核处理团队）
   → 预填"七点模板"，实地家访时填入实际反馈
   → 提交
   → 列表/详情中可一键预览打印 / 导出 PDF
   → 每月/每季批量导出 Excel 给部门归档
```

### 7.3 审计追溯流程

```
管理员 → /admin/logs
   → 按时间倒序查看
   → 筛某用户、筛某动作
   → 异常操作可直接定位 IP、操作时间、详情描述
```

## 八、运行与部署

### 8.1 开发环境

```bash
# 后端
cd backend
pip install -r requirements.txt
python init_db.py                 # 首次建库 + 默认管理员
python -m uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# 前端
cd frontend
npm install
npm run dev                       # 默认 http://localhost:5173
```

### 8.2 生产部署（内网单机）

1. 后端：gunicorn + uvicorn（或直接 uvicorn 绑定内网 IP），Nginx 反向代理
2. 前端：`npm run build`，把 `dist/` 放到 Nginx 或后端静态目录
3. 数据库：默认 SQLite 文件 `digital_portrait.db`；**改 `backend/app/config.py` 的 `DATABASE_URL` 即可迁移到 SQL Server**

### 8.3 默认账号

| 字段 | 值 |
|------|------|
| EHR 号 | `0000001` |
| 密码 | `1234567` |
| 角色 | 管理员 |
| 登录后 | 立即改密 |
