# 家访记录功能 - 需求梳理与实现方案

## 一、需求梳理

### 1.1 权限逻辑

| 角色 | 家访记录操作权限 |
|------|------------------|
| **组长 (leader)** | 可对名下组员进行家访并填写/编辑家访记录；可查看组员的家访记录 |
| **普通员工 (user)** | 仅可查看自己的家访记录（只读） |
| **管理员 (admin)** | 可查看所有人的家访记录；可编辑任意记录（可选，或仅组长可编辑） |

> **说明**：建议仅组长可新建/编辑家访记录，管理员仅查看；若需管理员也可编辑，可在实现时调整。

### 1.2 数据组成

家访记录包含两部分，合并为一条记录存储：

#### 家访明细信息（列表/概览用）

| 字段 | 说明 |
|------|------|
| 被家访人 EHR 号 | 关联用户 |
| 岗位 | 被家访人岗位 |
| 联系电话 | 被家访人手机号 |
| 家访年度 | 如 2024 |
| 家访时间 | 具体日期时间 |
| 家访方式 | 线上 / 线下 |
| 家访地址 | 实际家访地点 |
| 家访人员及岗位 | 执行家访的人员姓名及岗位（文本） |
| 是否已家访 | 布尔值，区分计划/已完成 |

#### 家访记录表（完整表单，可导出）

| 字段 | 说明 |
|------|------|
| 团队名称 | 被家访人所在团队（来自 group_name） |
| 家访日期 | 与家访时间一致 |
| 被家访人 | 姓名 |
| 岗位 | 同上 |
| 地址 | 家庭地址 |
| 手机号 | 同上 |
| 家庭电话 | 座机等 |
| 家属 1 信息 | 姓名、关系、联系方式、工作单位 |
| 家属 2 信息 | 同上 |
| 员工家庭情况及家属反馈意见 | 大段文本，可含七点模板 |

## 二、数据库设计

### 2.1 表结构：`home_visit_records`

```sql
CREATE TABLE home_visit_records (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    visited_user_id INTEGER NOT NULL,           -- 被家访人 user_id (FK)
    visitor_user_id INTEGER NOT NULL,           -- 家访人（组长）user_id (FK)
    
    -- 家访明细
    visit_year INTEGER NOT NULL,                 -- 家访年度
    visit_time DATETIME NOT NULL,                -- 家访时间
    visit_method VARCHAR(20) NOT NULL,           -- 线上/线下
    visit_address VARCHAR(500),                  -- 家访地址
    visitor_info VARCHAR(500),                  -- 家访人员及岗位
    is_visited BOOLEAN DEFAULT FALSE,            -- 是否已家访
    
    -- 家访记录表（与明细重叠的从档案/用户表可带出，此处存实际填写值）
    team_name VARCHAR(100),                      -- 团队名称
    visit_date DATE,                             -- 家访日期（冗余，便于导出）
    position VARCHAR(200),                       -- 岗位
    contact_phone VARCHAR(20),                  -- 联系电话
    address VARCHAR(500),                       -- 地址
    mobile VARCHAR(20),                         -- 手机号
    home_phone VARCHAR(20),                     -- 家庭电话
    
    -- 家属 1
    family1_name VARCHAR(100),
    family1_relation VARCHAR(50),
    family1_contact VARCHAR(50),
    family1_work_unit VARCHAR(300),
    
    -- 家属 2
    family2_name VARCHAR(100),
    family2_relation VARCHAR(50),
    family2_contact VARCHAR(50),
    family2_work_unit VARCHAR(300),
    
    -- 反馈意见
    feedback TEXT,                               -- 员工家庭情况及家属反馈意见
    
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

- `visited_user_id`：被家访人，用于权限过滤（本人/组长看组员/管理员看全部）
- `visitor_user_id`：记录由谁创建，便于审计
- 岗位、联系电话等可从 Profile/ContactInfo 带出，也可在表单中覆盖保存

## 三、后端 API 设计

### 3.1 路由前缀

`/api/home-visits`

### 3.2 接口列表

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | `/` | 分页列表（家访明细） | 本人/组长看组员/管理员看全部 |
| GET | `/{id}` | 单条详情（含完整记录表） | 同上 |
| POST | `/` | 新建家访记录 | 仅组长（且被家访人须为本组成员） |
| PUT | `/{id}` | 更新家访记录 | 仅组长（且记录属于本组） |
| DELETE | `/{id}` | 删除家访记录 | 仅组长或管理员 |
| GET | `/export/{id}` | 导出单条为 PDF/打印页 | 有查看权限即可 |

### 3.3 列表查询参数

- `page`, `page_size`
- `visited_ehr`：按被家访人 EHR 筛选
- `visit_year`：按家访年度筛选
- `is_visited`：是否已家访
- `visit_method`：家访方式

### 3.4 权限校验逻辑

复用现有 `_can_access(viewer, target)` 思路：

- 管理员：可访问所有记录
- 组长：仅可访问 `visited_user.group_name == viewer.group_name` 的记录
- 普通员工：仅可访问 `visited_user_id == viewer.id` 的记录

新建时：组长只能为 `group_name` 相同的组员创建记录。

## 四、前端 UI 实现方案

### 4.1 路由与菜单

| 路径 | 页面 | 访问权限 |
|------|------|----------|
| `/home-visits` | 家访记录列表 | 所有登录用户 |
| `/home-visits/new` | 新建家访记录 | 仅组长 |
| `/home-visits/:id` | 家访记录详情/编辑 | 按权限 |

**侧边栏菜单**：在「首页」与「我的档案」之间增加「家访记录」入口，所有角色可见。

### 4.2 页面结构

#### 4.2.1 家访记录列表页 (`/home-visits`)

- **布局**：Card + Table，风格与「档案管理」一致
- **筛选**：家访年度、家访方式、是否已家访、被家访人（EHR/姓名）
- **表格列**：
  - 被家访人（姓名 + EHR）
  - 岗位
  - 家访年度
  - 家访时间
  - 家访方式（Tag：线上/线下）
  - 是否已家访（Tag：已家访/未家访）
  - 操作：查看、编辑（组长）、导出、删除（组长/管理员）
- **新建按钮**：仅组长可见，跳转 `/home-visits/new`
- **权限**：普通员工仅看到自己的记录；组长看到本组；管理员看到全部

#### 4.2.2 新建/编辑页 (`/home-visits/new` 或 `/home-visits/:id/edit`)

采用 **Steps 或 Tabs** 分步填写，减少单页信息量：

**第一步：家访明细**

- 被家访人：下拉选择（组长仅能选本组成员，从档案列表接口带出）
- 岗位、联系电话：选择被家访人后自动带出，可修改
- 家访年度：年份选择器
- 家访时间：DatePicker + TimePicker 或 DateTimePicker
- 家访方式：单选（线上/线下）
- 家访地址：Input
- 家访人员及岗位：Input（可填多人）
- 是否已家访：Switch

**第二步：家访记录表**

- 团队名称：自动带出 group_name，可改
- 家访日期：DatePicker
- 被家访人、岗位、地址、手机号、家庭电话：第一步或档案带出，可编辑
- 家属 1：姓名、关系、联系方式、工作单位（4 个 Input）
- 家属 2：同上
- 员工家庭情况及家属反馈意见：TextArea（可预填七点模板，见下）

**七点模板（可选预填）**：

```
（一）家庭主要成员情况：……
（二）员工八小时以外动向：……
（三）员工思想动态：……
（四）家庭经济状况：……
（五）家庭关系情况：……
（六）员工家庭生活实际困难：……
（七）总体评价及跟进意见：……
```

#### 4.2.3 详情/只读页 (`/home-visits/:id`)

- 普通员工：只读展示
- 组长：可进入编辑或直接展示，顶部有「编辑」「导出」按钮
- 管理员：只读 + 导出

布局与表单一致，以只读形式展示两步内容。

### 4.3 导出/打印功能

**目标**：生成与纸质「集约运营中心（广东）员工家访记录表」布局一致的表格。

**实现方式**：

1. **方案 A（推荐）**：前端用 HTML + CSS 精确还原表格布局，通过 `window.print()` 或 `html2pdf.js` 生成 PDF
2. **方案 B**：后端用 ReportLab / WeasyPrint 等生成 PDF，接口返回文件流

**推荐方案 A** 的原因：

- 与现有 React 技术栈一致
- 样式可复用 Ant Design 风格
- 无需新增 Python 依赖

**导出页结构**：

- 新建隐藏的打印专用 DOM，包含：
  - 标题：集约运营中心（广东）员工家访记录表
  - 基本信息表格（团队名称、家访日期、被家访人、岗位、地址、手机号、家庭电话）
  - 家属信息表格（家属 1、家属 2 各 4 列）
  - 员工家庭情况及家属反馈意见（大块文本）
  - 家访人姓名、签字区
- 调用 `window.print()` 或使用 `html2pdf.js` 导出 PDF

**依赖**：可新增 `html2pdf.js` 或 `jspdf` + `html2canvas`，或仅用 `window.print()` 实现打印。

## 六、实现状态（已确认并实现）

1. **管理员编辑权限**：管理员不可编辑，仅组长可编辑；组长可删本组记录，管理员可删任意记录。
2. **导出格式**：支持「预览打印」和「导出 PDF」。
3. **家访人签字**：留空，由线下签字。
4. **团队名称**：固定为「审核处理团队」，不可修改。
